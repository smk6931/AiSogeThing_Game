# Title: Workflow Rules
Description: Claude와 Codex가 공통으로 따라야 하는 작업 시작, 계획, 구현, 검증, 리뷰 흐름을 정의한다.
When-To-Read: 새 기능 구현, 구조 변경, 버그 수정, 문서 정리, 검증 절차 정리가 필요한 모든 작업 전
Keywords: workflow, process, plan, implement, verify, review, handoff
Priority: high

## 목적

- 프로젝트 공통 작업 절차를 `agents/` 안에서 단일 source of truth로 유지한다.
- `CLAUDE.md`, `CODEX.md`에는 툴별 사용법만 남기고 공통 작업 방식은 여기서 관리한다.

## 기본 원칙

- 작업 전에는 항상 유저 요청, 수정 대상 파일, 실행 명령 목적을 먼저 정리한다.
- 관련 있는 `agents/` 문서만 읽고 시작한다.
- 관련 규칙이 없다면 루트 문서에 임시로 쓰지 말고 `agents/` 안에 새 문서로 추가한다.
- 큰 작업은 계획을 먼저 세우고, 구현 후에는 검증을 수행한다.
- 공통 설계 규칙은 `AGENTS.md`, `CLAUDE.md`, `CODEX.md`에 중복 작성하지 않는다.

## 권장 작업 흐름

### 1. Analyze

- 유저 목표를 한두 문장으로 다시 정리한다.
- 수정 범위와 건드리면 안 되는 범위를 구분한다.
- 필요한 `agents/` 문서를 고른다.

### 2. Plan

아래 중 하나라도 해당하면 계획을 먼저 세운다.

- 3개 이상 파일 수정
- DB 스키마, 월드 구조, API 구조 변경
- 렌더링 구조나 상태관리 구조 변경
- 새 시스템 추가
- 요구사항이 애매함

계획에는 아래가 포함되어야 한다.

- 목표
- 수정 파일과 영향 범위
- 구현 순서
- 주요 리스크
- 검증 방법

### 3. Implement

- 최소 범위 수정부터 시작한다.
- 기존 코드베이스 패턴을 우선 따른다.
- 공통 규칙을 우회하기 위해 루트 툴 문서에 새 규칙을 쓰지 않는다.

### 4. Verify

아래 시점에는 검증을 우선한다.

- 구현 완료 직후
- 커밋 전
- 빌드, 런타임, 네트워크, DB 관련 변경 후

가능한 검증 예:

- build
- type check
- lint
- test
- 수동 시나리오 확인

검증이 불가능하면 이유를 명시한다.

### 5. Review

아래 작업은 리뷰 기준으로 한 번 더 본다.

- 보안 관련 변경
- API, DB 변경
- 에이전트가 대량 생성한 코드
- 리팩터링으로 영향 범위가 큰 변경

리뷰에서는 아래를 우선 본다.

- 동작 회귀
- 데이터 무결성
- 보안과 권한
- 테스트 누락
- 과한 리팩터링

## 문서 역할 분리

- `AGENTS.md`: 공통 규칙 진입점과 우선순위
- `CLAUDE.md`: Claude 전용 명령과 사용법
- `CODEX.md`: Codex 전용 명령과 사용법
- `agents/`: 공통 규칙 본문

## 금지 패턴

- 같은 공통 규칙을 `AGENTS.md`, `CLAUDE.md`, `CODEX.md`에 복붙하기
- 프로젝트 설계 규칙을 사용자 홈 전역 설정에 쓰기
- 툴별 편의 기능 때문에 공통 규칙을 무시하기
