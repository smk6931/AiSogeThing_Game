# Title: Documentation Writing Rules
Description: 사용자가 문서 작성을 요청했을 때 `docs/` 안의 적절한 카테고리를 고르고, 오늘 날짜 기준으로 파일을 작성하는 규칙.
When-To-Read: 문서 초안 작성, 계획 문서 정리, 작업 기록 작성, 완료 보고서 작성, 월드 설계 문서 추가 요청을 받았을 때
Keywords: docs, documentation, writing, category, date, plan, task, complete, world, capture
Priority: high

## 목적

- 문서 요청을 받으면 루트나 임의 위치가 아니라 `docs/` 아래 정해진 카테고리에 작성한다.
- 같은 성격의 문서가 같은 위치와 같은 날짜 규칙을 따르도록 유지한다.
- 에이전트가 문서 요청을 받았을 때 바로 작성 가능한 기본 절차를 제공한다.

## 기본 원칙

- 사용자가 "문서 써줘", "계획 정리해줘", "완료 문서 남겨줘"처럼 문서 작성을 요청하면 이 규칙을 먼저 따른다.
- 먼저 `docs/` 하위 카테고리 중 요청 목적과 가장 가까운 폴더를 고른다.
- 문서는 가능한 한 오늘 날짜를 포함해 저장한다.
- 기존 문서 관례가 있는 카테고리는 그 관례를 우선 따른다.
- 같은 내용을 `agents/`와 `docs/`에 중복 저장하지 않는다.
  - `agents/`는 규칙과 source of truth
  - `docs/`는 작업 기록, 계획, 설계, 완료 보고

## 카테고리 선택 규칙

- `docs/plan/`
  - 앞으로 할 구현 계획, 리팩터링 계획, 비교안, 단계별 실행안
- `docs/task/`
  - 당일 작업 메모, TODO 정리, 세부 작업 단위 기록
- `docs/complete/`
  - 완료 보고, 구현 결과 요약, 마감 기록
- `docs/world/`
  - 월드 구조, 지형, 레벨디자인, 파티션, 게임 규칙 설계
- `docs/capture/`
  - 스크린샷, 이미지 산출물, 레퍼런스 캡처

요청이 두 카테고리 사이에 걸치면 아래 우선순위를 따른다.

1. 월드/레벨 구조 자체를 정의하면 `docs/world/`
2. 아직 할 일을 설계하면 `docs/plan/`
3. 진행 중 작업 메모면 `docs/task/`
4. 끝난 결과 보고면 `docs/complete/`

## 날짜 규칙

- 오늘 날짜는 에이전트 실행 시점의 로컬 날짜를 기준으로 `YYYY-MM-DD` 형식으로 쓴다.
- `docs/plan/`과 `docs/task/`는 `docs/<category>/YYYY-MM-DD/` 폴더를 우선 사용한다.
- `docs/complete/`는 기존 관례에 맞춰 `YYYY-MM-DD_문서명.md` 형식 파일명을 우선 사용한다.
- `docs/world/`는 장기 참조 문서가 많으므로 날짜를 파일명 앞에 강제하지 않는다.
  - 다만 특정 일자의 설계안, 검토안, 비교안이면 `docs/plan/YYYY-MM-DD/` 아래에 둔다.
- 새 날짜 폴더가 필요하면 오늘 날짜로 새로 만든다.

## 작성 절차

1. 유저 요청 목적을 분류한다.
2. `docs/`에서 맞는 카테고리를 고른다.
3. 해당 카테고리의 기존 파일명 패턴을 확인한다.
4. 오늘 날짜 규칙에 맞는 경로를 정한다.
5. 문서를 작성한다.
6. 문서가 공통 규칙이 아니라 작업 산출물인지 다시 확인한다.

## 경로 예시

- 계획 문서: `docs/plan/2026-04-18/레벨디자인_방향_재검토.md`
- 작업 메모: `docs/task/2026-04-18/CityBlockOverlay_수정_메모.md`
- 완료 보고: `docs/complete/2026-04-18_월드_렌더링_수정_완료.md`
- 월드 설계: `docs/world/서울_레벨디자인_원칙.md`

## 금지 패턴

- 문서 요청인데 루트 폴더에 바로 `.md` 파일 생성
- 날짜 관례가 있는 카테고리에 날짜 없이 새 문서 생성
- 공통 규칙 문서를 `docs/`에 쓰기
- 작업 기록 문서를 `agents/`에 쓰기
