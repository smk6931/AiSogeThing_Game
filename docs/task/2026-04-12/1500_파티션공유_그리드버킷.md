# 파티션 공유 fetch + Grid Bucket 공간 인덱싱

## 작업 내용

1430 리팩토링 이후 남은 2개 추가 최적화 적용.

## 수정 파일

### 프론트엔드: 파티션 데이터 공유

**`front/src/entity/world/RpgWorld.jsx`**
- `worldApi` import 추가
- `sharedPartitions` state 추가
- `currentDong?.id` 변경 시 `getDongPartitions` 단일 fetch
- `PartitionBoundaryOverlay`, `GroupColorOverlay`에 `partitions={sharedPartitions}` prop 전달
- `currentDong` prop 제거 (두 컴포넌트가 더 이상 자체 fetch 하지 않으므로 불필요)

**`front/src/entity/world/PartitionBoundaryOverlay.jsx`**
- `useState`, `useEffect`, `worldApi` import 제거
- `currentDong` prop → `partitions` prop 으로 교체
- 자체 fetch 로직 완전 제거

**`front/src/entity/world/GroupColorOverlay.jsx`**
- 동일: `useState`, `useEffect`, `worldApi` import 제거
- `currentDong` prop → `partitions` prop 으로 교체
- 자체 fetch 로직 완전 제거

**효과**: 같은 동에서 두 컴포넌트가 각자 하던 `getDongPartitions` 호출이 RpgWorld의 단일 호출로 통합.

### 백엔드: Grid Bucket 공간 인덱싱

**`back/monster/managers/MonsterManager.py`**
- `_GRID_CELL = 30.0` 상수 추가 (감지 범위 25m보다 약간 크게)
- `_build_player_grid(player_list)` 함수 추가: 플레이어를 O(n)으로 그리드 버킷에 분류
- `_nearby_players(grid, mx, mz)` 함수 추가: 몬스터 위치 기준 인접 9셀만 반환
- game_loop에서 `player_list` 브루트포스 → `player_grid` + `_nearby_players` 교체

**효과**:
- 이전: 몬스터 수 × 플레이어 수 = O(n×m) 매 루프
- 이후: 그리드 구성 O(n) + 몬스터당 인접 셀 플레이어만 탐색 ≈ O(n + m×k) (k = 셀당 평균 플레이어 수, 보통 1~3)
- 플레이어 20명이 넓게 분포 시: 400회 → ~20회 거리 계산

## 검증 방법
- 서버 재시작 후 몬스터가 정상적으로 플레이어를 감지하고 이동하는지 확인
- 브라우저에서 `PartitionBoundaryOverlay`/`GroupColorOverlay` 토글 시 Network 탭에 API 요청 없어야 함
