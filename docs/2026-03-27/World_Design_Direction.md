# 2026-03-27 World Design Direction

## 왜 이 문서를 남기나

현재 고민의 핵심은 다음 두 가지다.

1. 이 프로젝트의 가장 중요한 우선순위인 "주변 환경 월드 디자인"을 어떤 방식으로 구축할 것인가
2. 지금 진행 중인 "서울 지형을 동/도로/섹션별로 쪼개서 바닥 텍스처를 입히는 방식"이 1단계 접근으로 맞는가

결론부터 적으면:

- `동/도로/섹션 단위로 쪼개서 바닥 텍스처를 입히는 방식`은 1단계로 맞다.
- 하지만 이것을 `최종 월드 제작 방식`으로 고정하면 안 된다.
- 이 작업은 어디까지나 `월드 시각 블록아웃 + 플레이 가능 영역 분류 + 스트리밍 단위 실험` 단계로 봐야 한다.

## 월드 꾸미는 사람을 뭐라고 부르나

보통 역할은 아래처럼 나뉜다.

- `Level Designer`
  - 플레이 동선, 전투/파밍/탐험 흐름, 위험도, 이벤트 배치 설계
- `Environment Artist`
  - 바닥, 건물, 오브젝트, 자연물, 라이팅, 색감 등 실제 시각 자산 제작
- `World Builder`
  - 지형, 오브젝트, 포인트 오브 인터레스트(POI)를 맵 위에 실제로 배치하는 실무 역할
- `Technical Level Designer` 또는 `Technical World Designer`
  - 월드 스트리밍, 로딩 단위, 충돌, 최적화, 데이터 구조까지 같이 만지는 역할

이 프로젝트는 서울 실지형 + 소셜 + 모바일 + 오픈월드 성격이 섞여 있으므로, 지금 가장 필요한 역할은 하나로 보면 `Technical World Designer`에 가깝다.

즉, 단순히 예쁘게 꾸미는 사람보다:

- 실제 서울 지형을 플레이 가능한 형태로 재구성하고
- 월드 데이터를 잘게 나누고
- 모바일에서 돌아가게 만들고
- 소셜/파밍/이벤트가 일어날 구조를 설계하는 역할이 더 중요하다

## 지금 텍스처 분할 방식은 1단계로 맞는가

### 답

`맞다.`

다만 전제는 분명하다.

- 이건 `최종 아트 파이프라인`이 아니라
- `월드 구조를 잡기 위한 1차 블록아웃/실험용 파이프라인`이다

### 왜 1단계로 맞는가

지금 단계에서 구역을 나누고 바닥을 입혀보는 작업은 아래 검증에 매우 유효하다.

- 플레이어가 서울 지형을 "구역"으로 인식하는지
- 도로/블록/녹지/수변이 게임 플레이에서 읽히는지
- 어느 크기로 쪼개야 로딩과 수정이 감당되는지
- 파밍/몬스터/이벤트 스폰 단위를 어디까지 세분화해야 하는지
- 구/동/도로가 실제로 재미 구조로 연결될 수 있는지

즉 지금 하는 작업은 헛수고가 아니라, 월드 시스템의 최소 단위를 찾는 중요한 실험이다.

## 하지만 이 방식을 최종 구조로 삼으면 안 되는 이유

문제는 `큰 바닥 텍스처를 잘게 나눠 붙이는 방식만으로는` 다음이 어렵다는 점이다.

- 지역 상태 변화
- 계절/시간/이벤트 연출
- 지역별 개성 강화
- 자원/몬스터/오브젝트 동적 교체
- 방송용 볼거리 연출
- 모바일 최적화

즉, 큰 그림 텍스처는 "보여주기"는 되지만 "운영하기 좋은 월드"는 아니다.

## 권장 월드 구조

서울 월드를 아래 계층으로 나누는 것이 좋다.

### 1. 구 (`District`)

- 목적: 큰 테마/경제권/위험도/문화권
- 예:
  - 동작구 = 한강변 + 생활권 + 밤 이벤트
  - 마포구 = 공연/버스킹/소셜 방송권
  - 강남구 = 거래/희귀 상점/고위험 고보상

### 2. 동 (`Dong`)

- 목적: 실제 스트리밍/로딩 단위
- 플레이어 근처 동만 로드하는 기준점
- 현재 구조에서 가장 현실적인 중간 단위

### 3. 섹터 또는 블록 (`Sector` / `Block`)

- 목적: 실제 바닥 텍스처, 채집물, 몬스터, 이벤트 배치 단위
- 도로망/지형/용도 기반으로 쪼개기
- "여기서 무슨 일이 일어나는가"를 결정하는 실전 단위

### 4. 포인트 오브 인터레스트 (`POI`)

- 목적: 방송각/집결/이벤트/상호작용 포인트
- 예:
  - 포장마차
  - 지하철 출구
  - 편의점
  - 공원
  - 다리 아래
  - 노량진 학원가
  - 홍대 공연장

결국 플레이어는 텍스처보다 POI에서 기억을 만든다.

## 지금 단계에서 맞는 실무 방향

지금은 아래 순서가 맞다.

### Step 1. 바닥 분할 유지

- 동/도로/블록 기반으로 바닥을 계속 나눈다
- 단, "한 장의 완성 아트"를 만들려 하지 말고
- `구역 식별용`, `동선 확인용`, `스폰 테스트용` 수준으로 유지한다

### Step 2. 텍스처보다 데이터 구조를 먼저 정리

- 각 섹터에 아래 메타데이터를 붙일 수 있어야 한다
  - `district_id`
  - `dong_id`
  - `sector_id`
  - `biome_type`
  - `zone_type`
  - `road_access`
  - `event_spawn_weight`
  - `resource_spawn_table`
  - `monster_spawn_table`
  - `social_score`

이게 있어야 나중에 운영형 콘텐츠가 붙는다.

### Step 3. 바닥 표현을 레이어 방식으로 바꾸기

최종적으로는 아래 구조가 좋다.

- base terrain color
- road/path overlay
- district mood overlay
- zone/landuse overlay
- event decal overlay
- object/prop placement

즉 "텍스처 1장"보다 "레이어 조합"이 운영에 강하다.

### Step 4. 플레이 포인트를 먼저 만든다

월드를 꾸민다는 건 결국 "여기서 뭘 하게 만들 것인가"다.

그래서 각 동마다 최소 3종은 있어야 한다.

- 파밍 포인트
- 소셜 포인트
- 이벤트 포인트

예:

- 파밍 포인트: 골목, 공원, 하천변
- 소셜 포인트: 광장, 편의점, 공연장 앞
- 이벤트 포인트: 다리 아래, 지하철역, 야시장

## 제일 중요한 판단

현재 방식이 맞냐는 질문에 대한 가장 정확한 답은 이거다.

- `시작점으로는 맞다`
- `완성 방식으로는 부족하다`

정확히는 아래처럼 이해해야 한다.

- 지금 하는 분할 텍스처 작업 = 월드 블록아웃
- 이후 필요한 것 = 레이어화, 메타데이터화, 스트리밍화, POI화

## 서버 운영 관점에서의 판단

나중에 서버 운영까지 생각하면, `구 단위 큰 자산`보다 `동/섹터 단위 조립형 구조`가 훨씬 안전하다.

이유:

- 필요한 부분만 로딩 가능
- 수정 범위가 작음
- 특정 지역만 업데이트 가능
- 이벤트 지역만 별도 교체 가능
- 모바일 메모리 사용량 통제 가능
- 플레이어 밀집 지역만 따로 최적화 가능

## 지금 당장 추천하는 다음 작업

1. 구 전체를 예쁘게 만들려 하지 말고 `핵심 1개 동`만 완성도 있게 테스트한다
2. 그 동 안에서 섹터를 3~5개 타입으로 분리한다
3. 각 섹터에 바닥 텍스처만이 아니라 `역할`을 부여한다
4. 같은 동에 `파밍 1개`, `소셜 1개`, `이벤트 1개`를 박아본다
5. 이 구조가 재미가 나오면 그때 인접 동으로 확장한다

## 최종 메모

이 프로젝트에서 월드 디자인의 본질은 `서울을 똑같이 그리는 것`이 아니다.

본질은:

- 서울이라는 현실감을 유지하면서
- 플레이 가능한 단위로 재구성하고
- 사람끼리 만나고
- 파밍하고
- 방송 콘텐츠가 터지는 장소를 만드는 것이다

그러므로 지금의 바닥 분할 작업은 버릴 방향이 아니라, `더 구조화해서 다음 단계로 넘겨야 하는 올바른 1단계 작업`이다.
