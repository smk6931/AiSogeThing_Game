# 파티션 그룹 텍스처 생성 방식 분석 및 레이어 재배치 계획

**작성일**: 2026-04-11  
**대상**: world_partition 그룹핑 시스템 + ComfyUI 텍스처 파이프라인 + 렌더링 레이어 재배치

---

## 1. 현황 분석 리포트 — 실제 구현 vs 설명된 아키텍처

### 1-1. 사용자가 설명한 아키텍처

| 구성 요소 | 설명 |
|---|---|
| `world_partition` | 가장 작은 단위 파티션 |
| `world_partition_group` | 그룹 이름 + 페르소나 저장 |
| `world_partition_group_member` | 두 테이블 간 참조(junction) 테이블 |
| ComfyUI 텍스처 | 파티션 polygon 꼭지점을 추출 → ComfyUI로 바닥 텍스처 생성 |
| 그룹 페르소나 | group 단위로 공통 페르소나/테마 부여, 내부 파티션은 세부 변형 |
| 그룹 경계 | 파티션들이 최대한 뭉쳐 보이도록 경계 설정 |

---

### 1-2. 실제 구현 상태

#### DB 스키마 — `back/world/models/models.py`

실제 테이블은 **`world_level_partition` 단일 테이블**에 group 정보가 컬럼으로 내장:

```
world_level_partition
├── partition_key       (파티션 고유 키)
├── group_key           (그룹 식별자 — 외래키 X, 내장 컬럼)
├── group_seq           (그룹 내 순서)
├── group_display_name  (그룹 표시 이름)
├── group_theme_code    (테마 코드)
├── persona_tag         (페르소나 — 파티션 레벨)
├── texture_profile     (텍스처 프로필 — 파티션 레벨)
├── boundary_geojson    (GeoJSON polygon 꼭지점 — 렌더링/포함 판단용)
└── ...
```

**world_partition_group** 테이블: **존재하지 않음**  
**world_partition_group_member** 참조 테이블: **존재하지 않음**  
→ 설계 문서(`agents/db/partition_grouping_concept.md`)에 의도적으로 단일 테이블로 통합한 것으로 기록됨

#### persona_tag 위치

- 현재: `world_level_partition.persona_tag` (파티션 레벨 컬럼)
- 설명된 아키텍처: 그룹 레벨에서 공통 페르소나 부여
- **차이**: 그룹 공통 페르소나가 group_theme_code로 대체되어 있으며, 파티션별 세부 persona_tag는 있으나 그룹 단위 집계/분리 구조는 없음

#### ComfyUI 텍스처 파이프라인

- `/tools/ComfyUI/` 디렉토리 존재 (ComfyUI 설치 확인)
- polygon 꼭지점 → ComfyUI 워크플로우 연결: **미구현**
- 현재 텍스처 렌더링 방식:
  - `DongGroundMesh.jsx`: 절차적 캔버스 기반 텍스처 (코드 내 생성)
  - `CityBlockOverlay.jsx`: `/front/public/ground/` 폴더 기반 이미지 파일 로드
  - 텍스처 인덱스 결정: `hash(group_key|group_theme_code|texture_profile|partition_seq)`

#### 렌더링 레이어 현황 — `PartitionBoundaryOverlay.jsx` + `CityBlockOverlay.jsx`

| 레이어 이름 | 변수/prop | renderOrder | 색상 | 현재 상태 |
|---|---|---|---|---|
| 그룹텍 (Group Texture) | `showCurrentGroupTexture` | — | 테마별 | CityBlockOverlay.jsx에서 처리 |
| 미세선 (Micro Boundary) | `showMicroBoundaries` | 13 | `#f7d56b` opacity 0.24 | 구현됨 |
| 그룹선 (Group Boundary) | `showGroupBoundaries` | 14 | `#8bd8ff` / `#67e8d6` | 구현됨 (엣지 중복 제거로 도출) |
| 강조 (Highlight) | `highlightCurrentGroup` | 15–16 | `#67e8d6` / 파티션 하이라이트 | 구현됨 |

그룹 경계선은 **인접 파티션 간 공유 엣지 제거 알고리즘**으로 도출 (별도 group boundary polygon 없음).

### 1-3. 차이 요약표

| 항목 | 설명된 아키텍처 | 실제 구현 | 차이 |
|---|---|---|---|
| DB 구조 | 3개 테이블 (partition / group / member) | 1개 테이블 (컬럼 내장) | 단순화됨 (의도적) |
| 그룹 페르소나 | group 레벨 별도 저장 | group_theme_code 컬럼으로 내장 | 구조는 다르나 동일 목적 |
| ComfyUI 연결 | polygon → ComfyUI → texture 파이프라인 | 미연결, 폴더 기반 이미지 사용 | **미구현** |
| 바닥 텍스처 | ComfyUI 생성 이미지 | 절차적 canvas + 폴더 이미지 | **미구현** |
| 그룹 경계 | 별도 polygon 저장 추정 | 파티션 엣지 중복 제거 알고리즘 | 구현 방식 다름 |

## 2. ComfyUI + Polygon 꼭지점 기반 텍스처 생성 방식 평가

### 2-1. 방식의 강점

| 강점 | 설명 |
|---|---|
| 지형 형태 반영 | polygon 꼭지점 형태를 그대로 마스킹에 활용 → 경계가 자연스러움 |
| 페르소나 일관성 | group_theme_code + persona_tag를 프롬프트에 넣으면 그룹 내 통일감 |
| 확장성 | ComfyUI API로 자동화 가능, batch 처리 가능 |
| 시각적 다양성 | 같은 그룹 안에서도 파티션마다 세부 변형 가능 |

### 2-2. 방식의 리스크

| 리스크 | 수준 | 설명 |
|---|---|---|
| 생성 비용 | 높음 | 파티션 수가 많으면 ComfyUI 생성 시간 + 저장 공간 급증 |
| 경계 이음매 | 중간 | 인접 파티션 간 텍스처가 자연스럽게 이어지지 않을 수 있음 |
| polygon→UV 매핑 | 중간 | 불규칙 polygon을 텍스처 UV에 맞추는 작업 필요 |
| 런타임 로딩 | 낮음 | 텍스처 파일 수가 늘어나면 초기 로딩 부담 |
| 파이프라인 복잡도 | 높음 | ComfyUI 워크플로우 관리 + API 연동 + 파일명 규칙 합의 필요 |

### 2-3. 종합 평가

**이 게임에 적합한 방식인가? → 적합하나 단계적 접근 필요**

- 현재 규모에서는 폴더 기반 이미지 + hash 인덱스 방식으로도 충분
- ComfyUI 파이프라인은 **오프라인 사전 생성(offline pre-generation)** 방식이 현실적
  - 런타임에 ComfyUI 호출 X → 빌드/배포 전 텍스처를 일괄 생성하고 `/public/ground/`에 배치
- polygon 꼭지점 기반 마스킹은 **그룹 경계 텍스처**에만 적용하고, 파티션 내부는 타일 반복 텍스처로 처리하는 하이브리드가 권장

## 3. 파티션 그룹 재배치 시 영향받는 레이어 및 체크리스트

### 3-1. 재배치 시 영향받는 레이어

파티션 그룹 재배치 = `group_key`, `group_seq`, `group_display_name`, `group_theme_code` 변경

| 레이어 | 컴포넌트 | 영향 내용 |
|---|---|---|
| **그룹텍** | `CityBlockOverlay.jsx` | group_key 변경 → 텍스처 인덱스 hash 변경 → 다른 텍스처 로드 |
| **그룹선** | `PartitionBoundaryOverlay.jsx` | 그룹 경계선이 새 group_key 기준으로 재계산됨 |
| **강조** | `PartitionBoundaryOverlay.jsx` | currentGroup 하이라이트가 새 group_key 기준으로 변경 |
| **미세선** | `PartitionBoundaryOverlay.jsx` | partition 자체 경계이므로 그룹 재배치 영향 없음 |
| **포탈/지역** | `onPortalEncounter`, `currentRegionInfo` | currentMapId 변경 시 영향, group 변경만으로는 무관 |
| **자동 이동** | `autoFarmRange`, RpgWorld | 그룹 경계 기반 AI가 있다면 영향, 현재는 무관 |

## 4. 구현 체크리스트

### Phase 0 — DB 스키마 정렬 (선행 조건)

- [ ] `world_level_partition`에 `group_persona` 컬럼 추가 여부 결정
  - 현재 `persona_tag`가 파티션 레벨에만 있음
  - 그룹 공통 페르소나가 필요하면 `group_theme_code` 활용 or 신규 컬럼 추가
- [ ] 그룹 재배치 시 migration 스크립트 준비
  - `UPDATE world_level_partition SET group_key=?, group_seq=?, group_display_name=?, group_theme_code=? WHERE partition_key IN (...)`
- [ ] 재배치 후 group_key 중복/누락 검증 쿼리 작성

### Phase 1 — 그룹 경계 재계산 검증

- [ ] `PartitionBoundaryOverlay.jsx` 엣지 중복 제거 로직이 새 group_key 기준으로 올바르게 동작하는지 테스트
  - 경계 파티션이 같은 그룹으로 묶이면 경계선 사라짐 확인
  - 다른 그룹으로 분리되면 경계선 나타남 확인
- [ ] 그룹 경계 색상이 `#67e8d6` (현재 그룹) / `#8bd8ff` (타 그룹) 올바르게 적용되는지 확인
- [ ] renderOrder 우선순위 확인: 미세선(13) < 그룹선(14) < 강조(15) < 현재파티션(16)

### Phase 2 — 그룹 텍스처 재배치 반영

- [ ] `CityBlockOverlay.jsx`에서 텍스처 인덱스 hash 로직 확인
  - `hash(group_key|group_theme_code|texture_profile|partition_seq)` 기반
  - group_key 변경 시 자동으로 다른 텍스처 인덱스 계산됨
- [ ] `/front/public/ground/` 텍스처 파일 인덱스가 새 hash에 대응하는지 확인
- [ ] 재배치 후 텍스처 누락(빈 바닥) 파티션 없는지 전체 맵 순회 테스트

### Phase 3 — ComfyUI 텍스처 파이프라인 구축 (선택적, 중기)

- [ ] `boundary_geojson`에서 꼭지점 좌표 추출 스크립트 작성
  - 대상: `back/` Python 스크립트 or 별도 `tools/` 유틸
  - 출력: 각 파티션의 polygon 좌표 리스트 (lat/lon → 픽셀 좌표 변환 포함)
- [ ] ComfyUI API 연동 스크립트 작성
  - 입력: polygon 좌표 + group_theme_code + persona_tag
  - 워크플로우: ControlNet(마스크) + 텍스트 프롬프트(페르소나) → 텍스처 이미지
  - 출력: `/front/public/ground/{hash}.png` 파일 저장
- [ ] 그룹 내 파티션 간 이음매(seam) 처리 방법 결정
  - 옵션 A: 그룹 전체 영역을 하나의 큰 텍스처로 생성 후 파티션별 마스킹
  - 옵션 B: 파티션별 개별 생성 + 경계부 blending
- [ ] 오프라인 사전 생성 자동화 스크립트 (`tools/generate_textures.py` 등)
- [ ] 생성된 텍스처 파일 naming 규칙 정의 및 `CityBlockOverlay.jsx` 로딩 경로 연동

### Phase 4 — 강조(Highlight) 레이어 재배치 대응

- [ ] `highlightCurrentGroup` 로직이 새 group_key 기준 파티션 세트를 정확히 하이라이트하는지 확인
- [ ] 플레이어 현재 위치 → group_key 판단 로직 (`point-in-polygon` 기반) 검증
- [ ] 그룹 재배치 후 인접 그룹 탐색 700m 반경(`CityBlockOverlay.jsx`) 기준 유효성 확인

### Phase 5 — QA 및 검증

- [ ] 파티션 그룹 재배치 전/후 스크린샷 비교
- [ ] 그룹선(GroupBoundary)이 새 경계에서 올바르게 렌더링되는지 확인
- [ ] 미세선은 변경 없는지 확인 (partition 자체 경계 → 그룹 재배치 무관)
- [ ] 그룹텍 텍스처 로딩 에러 없는지 콘솔 확인
- [ ] 모바일 + 데스크탑 양쪽에서 렌더링 확인

## 5. 권장 실행 순서

```
Phase 0 (DB 정렬) → Phase 1 (경계 재계산 검증) → Phase 4 (강조 레이어 검증)
→ Phase 2 (텍스처 재배치 반영) → Phase 5 (QA)
→ (선택) Phase 3 (ComfyUI 파이프라인)
```

Phase 3 (ComfyUI)는 현재 폴더 기반 텍스처가 동작 중이므로 **별도 스프린트**로 분리 권장.
