# 3D 지형 고도 도입 계획

## 배경 및 목표

### 계기
- hordes.io 분석 결과, 웹에서도 3D 지형(하이트맵 기반)이 충분히 구현 가능함을 확인
- 현재 게임 월드는 완전 평면(y=0.55 고정) — 공간감과 지형 느낌이 없음
- 파티션 시스템이 OSM 경계 기반으로 이미 잘 쪼개져 있어 고도 도입의 기반이 됨

### hordes.io가 쓰는 기술 (참고)
- 하이트맵 PNG → Vertex Shader로 Y값 변위
- Fragment Shader `fract()` 로 등고선 표현
- 텍스처 스플래팅 (높이·경사도별 재질 블렌딩)
- 커스텀 WebGL 렌더러 (OGL 기반)

---

## 선택지 비교

### Option A — hordes.io 방식 전면 3D 지형
```
장점: 비주얼이 hordes.io급, smooth terrain
단점:
  - 플레이어 이동/충돌 전면 재설계
  - 몬스터 스폰 로직 변경
  - 카메라 terrain-follow 추가
  - 국토지리정보원 DEM 데이터 파이프라인 신규 구축
  - 파티션 polygon → 3D 지형 위 draping 새로 설계
  - 최소 2~3주 작업, 기존 코드 상당 부분 리셋
```

### Option B — 파티션 단위 계단식 고도 (채택)
```
장점:
  - CityBlockOverlay.jsx 최소 수정 (y-offset만 추가)
  - 플레이어/몬스터/카메라 코드 변경 없음
  - 실제 서울 DEM 평균값 사용 가능
  - v2 smooth terrain 전환 시 elevation 데이터 그대로 활용
단점:
  - Smooth terrain 아님 (계단식)
  - 등고선 표현 제한적
  - 파티션 경계에서 단차 발생
```

**결정: Option B 채택.** 현재 게임 시스템(전투, 스폰, 이동)과 충돌 없이 지형 느낌을 줄 수 있고, 나중에 v2로 자연스럽게 확장 가능.

## 구현 체크리스트

### Phase 1 — 데이터 준비
- [ ] 국토지리정보원 DEM 5m 해상도 데이터 다운로드 (서울 범위)
- [ ] DEM → 파티션별 평균 고도 계산 스크립트 작성
- [ ] `world_partition` 테이블에 `elevation_m` 컬럼 추가 (Alembic 마이그레이션)
- [ ] 파티션별 평균 고도값 DB 저장

### Phase 2 — 렌더링
- [ ] `CityBlockOverlay.jsx`: 파티션별 `elevation_m` 기반 y-offset 적용
- [ ] 인접 파티션 간 단차 완화 (높이 차 캡: 최대 Xm 이하로 제한)
- [ ] 플레이어/몬스터 위치의 현재 파티션 고도 쿼리 (선택)

### Phase 3 — 시각 개선 (선택)
- [ ] 파티션 경계 단차 처리: 경사면 mesh 추가 (optional)
- [ ] Fragment Shader 등고선 — `fract()` 기반 라인 (optional)
- [ ] 고도별 텍스처 재질 가중치 변화 (산지=rock, 저지대=grass)

## 주의사항

- 현재 `positions[vi++] = 0.55` (CityBlockOverlay) 이 값을 `0.55 + elevation` 으로 교체
- 고도 스케일은 게임 단위로 변환 필요: 실제 1m → 게임 world unit 비율 결정 필요
- 서울 고도 범위: 0m(한강) ~ 836m(북한산) — 게임에서는 과장 또는 압축 적용 고려
- 파티션이 없는 구역은 기본 y=0 유지
