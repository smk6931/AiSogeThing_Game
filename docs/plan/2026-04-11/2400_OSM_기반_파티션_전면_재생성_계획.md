## 배경 및 목표

현재 파티션은 손으로 그린 "Sector Plot" 임의 폴리곤을 도로로 자른 것.  
→ OSM 실제 지형 feature(공원, 강, 건물군, 캠퍼스)가 경계에 반영되지 않음.  
→ 도로가 공원을 가로지를 때 기묘한 모양의 파티션, 빵꾸 뚫린 형태 발생.

**목표**:  
Overpass API로 OSM 실제 폴리곤을 가져와 **도로 등급에 따라 스마트하게 경계를 결정**,  
이쁘게 정리된 파티션으로 전면 교체. 동별로 순차 처리.

## 핵심: 도로 등급별 분할 전략

### 도로가 자연 지형을 가로지를 때 처리 원칙

```
도로 등급           분할 여부        이유
────────────────────────────────────────────────
motorway/trunk    ★ 반드시 분할    고속화도로 = 완전한 경계
primary           ★ 반드시 분할    간선도로 = 서울 동 경계 기준
secondary         ★ 반드시 분할    2차선 이상 = 실질 생활권 경계
tertiary          △ 조건부 분할    나머지 fragment ≥ 3,000m²이면 분할
residential       △ 조건부 분할    나머지 fragment ≥ 2,000m²이면 분할
service/footway   ✕ 분할 안 함    내부 동선 = 공원/캠퍼스 유지
path/track        ✕ 분할 안 함    등산로/산책로 = 지형 무시
```

### "빵꾸 뚫린" 형태 처리

```
문제: 공원 한가운데로 도로가 지나가면 
      park polygon - road polygon = 구멍 뚫린 도넛 or MultiPolygon 조각

처리:
  1. primary 이상 → difference() → MultiPolygon으로 분리 → 각각 별도 파티션
  2. tertiary/residential → fragment < threshold이면 다시 merge
  3. service/footway → 아예 차감하지 않음 (공원 내부 산책로는 공원 그대로)

최종 확인: area_m2 < MIN_AREA(500m²)인 조각은 인접 파티션에 흡수
```

### 시각적 결과 예시

```
[기존 방식]                    [신규 방식]
  공원         road            공원A    road    공원B
┌─────────┐  │     │         ┌──────┐  │   │  ┌────┐
│  ██████ │  │     │    →    │ ████ │  │   │  │████│
│  █ 빵꾸 │  │ 도로 │         │      │  │   │  │    │
│  ██████ │  │     │         └──────┘  │   │  └────┘
└─────────┘  │     │         (두 파티션으로 깔끔하게 분리)
```

## OSM 레이어별 파티션 생성 전략

### 처리 우선순위 (레이어 점유 순서)

```
1. waterway relation/way (한강, 안양천, 도림천) → 수계 먼저 확정
2. 동 경계(dong boundary)                      → 클리핑 마스크
3. major highway buffer (primary+)             → 주요 경계선 확정
4. landuse / natural / leisure 폴리곤           → 지형 단위 클리핑
5. secondary+ highway로 지형 세분화             → tertiary 조건부
6. 빈 공간                                     → residential 기본
```

### OSM 태그 → theme_code 매핑

| OSM 태그 | dominant_landuse | theme_code | source_layer |
|---|---|---|---|
| `natural=water` / `waterway=river` | water | `ancient_waterway` | `osm_waterway` |
| `leisure=park` | park | `sanctuary_green` | `osm_park` |
| `natural=wood` / `landuse=forest` | forest | `sanctuary_green` | `osm_forest` |
| `landuse=residential` | residential | `residential_zone` | `osm_landuse` |
| `landuse=commercial` | commercial | `urban_district` | `osm_landuse` |
| `amenity=school/university` | educational | `academy_sanctum` | `osm_amenity` |
| `amenity=hospital` | medical | `sanctuary_healing` | `osm_amenity` |
| `highway=*` (폴리곤) | road | `urban_road` | `osm_road` |
| `landuse=military` | military | `fortress_grounds` | `osm_landuse` |
| `landuse=industrial` | industrial | `forge_district` | `osm_landuse` |

## 백업 전략 (DB 덤프)

### 방법: Docker pg_dump

```bash
# world_partition 관련 3개 테이블 데이터 덤프
docker exec game-sogething-db pg_dump \
  -U game_sogething -d game_sogething \
  -t world_partition \
  -t world_partition_group \
  -t world_partition_group_member \
  --data-only --inserts \
  > back/world/data/backups/partition_backup_YYYYMMDD_HHMM.sql
```

- **형식**: SQL INSERT 문 (복원 시 psql로 직접 실행)
- **위치**: `back/world/data/backups/` (gitignore 대상)
- **복원**: `docker exec -i game-sogething-db psql -U game_sogething -d game_sogething < backup.sql`

기존 JSON 백업보다 완전한 복원 보장. 각 동 처리 전 반드시 실행.

## 주의사항

- Overpass API rate limit: 쿼리 사이 2-3초 sleep, 결과는 캐시 파일로 저장
- 한강 폴리곤은 동 경계 밖으로 크게 나감 → Shapely `intersection(dong_boundary)` clip 필수
- 일부 OSM feature는 way(라인)만 있고 폴리곤 없음 → 스킵
- OSM relation 타입(공원, 강 등)은 outer/inner ring 조합 → shapely MultiPolygon으로 변환
- 기존 `world_partition_group` 21개 정의는 유지 → regroup이 새 파티션에 자동 배정
- `world_area.boundary_geojson`이 노량진 동 경계를 갖고 있어야 함 (미확인 시 확인 필요)
- 복원 명령: `docker exec -i game-sogething-db psql -U game_sogething -d game_sogething < backup.sql`
