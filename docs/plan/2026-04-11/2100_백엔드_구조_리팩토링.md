# 백엔드 구조 리팩토링 계획

**작성일**: 2026-04-11  
**상태**: 분석 완료 — 실행 미확정  
**목적**: 도메인별 레이어 불일치 해소, 유지보수성 향상

## 현재 문제 요약

### 1. 레이어 이름과 역할이 도메인마다 다름

| 도메인 | 실제 DB 쿼리 위치 | 실제 비즈니스 로직 위치 |
|---|---|---|
| user | `services/service.py` (이름은 service지만 repository 역할) | `services/auth.py` |
| player | `repository.py` (flat) | `managers/PlayerManager.py` + **router.py** |
| monster | 없음 (player.repository 간접 참조) | `managers/MonsterManager.py` |
| item | `repository.py` (flat) | `service.py` (flat) |
| world | `services/*.py` (service가 직접 쿼리) | `services/*.py` |

### 2. flat 파일 vs 폴더 혼재

```
item/repository.py      ← 파일
item/service.py         ← 파일
player/repository.py    ← 파일

vs

user/services/service.py    ← 폴더
world/services/             ← 폴더 (7개)
```

### 3. router.py 비대

- `player/routers/router.py` — 286줄
  - WebSocket 루프 전체 (position, chat, skill, hit_monster, use_item)
  - 몬스터 처치 → 보상 계산 → DB 저장 → 드롭 처리까지 한 함수 안에
  - 아이템 사용 로직도 router 안에서 직접 처리
- `user/routers/router.py` — 215줄 (허용 범위지만 인증 흐름 일부 포함)

### 4. 파일명 불일치

```
item/models/model.py      ← 단수
monster/models/model.py   ← 단수

vs

user/models/models.py     ← 복수
player/models/models.py   ← 복수
world/models/models.py    ← 복수
```

### 5. Manager 패턴이 player/monster에만 존재

- `PlayerManager`, `MonsterManager` — 싱글톤 인스턴스로 런타임 상태 유지
- 인메모리 상태 관리 + 비즈니스 로직 + 간접 DB 접근이 Manager 하나에 집중
- item, user, world에는 Manager 없음 → 상태 관리 기준 없음

## 목표 구조

### 레이어 기준 정의

```
router      HTTP/WebSocket 진입점만. 파라미터 추출 → service 호출 → 응답 반환
service     비즈니스 로직. DB를 직접 쓰지 않고 repository를 호출
repository  DB 쿼리만. SQL / ORM 코드 전담
manager     런타임 인메모리 상태 전담 (WebSocket 연결, 위치 캐시 등)
            DB는 repository를 통해서만 접근
schemas     Pydantic request/response 모델
```

## 변경 상세

### Phase 1 — 파일명 통일 (가장 안전, 즉시 가능)

> import 경로만 바뀌므로 로직 변경 없음

| 현재 | 변경 후 |
|---|---|
| `item/models/model.py` | `item/models/models.py` |
| `monster/models/model.py` | `monster/models/models.py` |
| `item/repository.py` | `item/repositories/repository.py` |
| `item/service.py` | `item/services/service.py` |
| `player/repository.py` | `player/repositories/repository.py` |

변경 시 함께 수정할 import:
- `item/routers/router.py` — `from item import repository`
- `item/services/service.py` — `from item import repository`
- `player/managers/PlayerManager.py` — `import player.repository as char_repo`
- `player/routers/router.py` — `import player.repository as char_repo`, `import item.repository as item_repo`
- `monster/models/model.py` import하는 곳 확인 후 수정

### Phase 2 — user/services/service.py 이름 정리

**현재 문제**: `user/services/service.py`가 이름은 service지만 내용은 순수 DB 쿼리만 있음

**변경**:
- `user/services/service.py` → `user/repositories/repository.py` (이름 + 폴더 변경)
- `user/services/service.py`는 새로 생성 — router에서 인증 흐름 조합 로직 이동

```python
# 변경 전: user/routers/router.py 안에 있던 것
user = await user_service.get_user_by_email(body.email)
auth_service.verify_password(...)
token = auth_service.create_access_token(...)

# 변경 후: user/services/service.py
async def login(email, password) -> dict:
    user = await user_repo.get_user_by_email(email)
    if not user or not auth.verify_password(password, user["password"]):
        raise ValueError("invalid credentials")
    return auth.create_access_token({"sub": str(user["id"])})
```

### Phase 3 — player/routers/router.py 분리 (핵심)

**현재**: 286줄 router에 WebSocket 루프 + 전투 로직 + 드롭 처리 혼재

**분리 목표**:

```
player/routers/router.py (목표: ~80줄)
    WebSocket accept/disconnect
    메시지 type 분기 → handler 호출

player/services/service.py (신규, ~120줄)
    handle_hit(user_id, monster_id, skill_name) → reward, drops
    handle_use_item(user_id, item_id) → heal result
    handle_move(user_id, position_data) → broadcast payload 구성
    (실제 DB 저장은 repository 호출)

player/repositories/repository.py (기존 player/repository.py 이동)
    get_character, save_character, get_ui_settings, save_ui_settings
```

**변경 전 (router.py 286줄 발췌)**:
```python

await char_repo.save_character(uid_int, reward_result["stats"])
dropped = await roll_drops(drop_table)
await grant_items_to_user(uid_int, dropped)
```

### Phase 4 — Monster/Player Manager 책임 축소

**현재 MonsterManager 책임 (341줄)**:
- 몬스터 스폰/위치 관리 (인메모리) ← **유지**
- 게임 루프 (`game_loop`) ← **유지**
- 전투 계산 (`handle_hit`, `_calc_damage`) ← **service로 이동**
- DB 조회 없음 (좋음) ← **유지**

**현재 PlayerManager 책임 (246줄)**:
- WebSocket 연결 관리 (인메모리) ← **유지**
- 브로드캐스트 (`broadcast`, `broadcast_nearby`) ← **유지**
- 스탯 계산 (`apply_damage`, `apply_heal`, `add_rewards`) ← **service로 이동**
- DB 저장 (`connect` 내부에서 `char_repo` 호출) ← **repository 직접 호출로 명시화**

### Phase 5 — world repository 분리

**현재**: `partition_service.py`가 직접 SQL 쿼리 실행

**목표**: DB 쿼리를 `repositories/partition_repository.py`로 분리

```python
# 현재: world/services/partition_service.py
return await fetch_all("SELECT id, ... FROM world_level_partition WHERE ...", ...)

# 목표: world/repositories/partition_repository.py
async def get_partitions_by_admin_area(admin_area_id: int) -> list[dict]:
    return await fetch_all("SELECT id, ... FROM world_level_partition WHERE ...", ...)

# world/services/partition_service.py
from world.repositories import partition_repository
async def get_partitions_by_dong_osm_id(dong_osm_id):
    area = await partition_repository.get_area_by_osm_id(dong_osm_id)
    return await partition_repository.get_partitions_by_admin_area(area["id"])
```

## 우선순위 및 리스크

| Phase | 작업 | 위험도 | 효과 |
|---|---|---|---|
| 1 | 파일명/경로 통일 | 낮음 (import만 수정) | 일관성 확보 |
| 2 | user service/repository 역할 정리 | 낮음 | 이름 혼란 제거 |
| 3 | player router 분리 | **높음** (WebSocket 로직 이동) | 유지보수성 대폭 향상 |
| 4 | Manager 책임 축소 | 중간 (게임 루프 관련) | 테스트 가능성 향상 |
| 5 | world repository 분리 | 낮음 | 계층 명확화 |

## 주의사항

- Phase 3은 WebSocket 실시간 로직이 얽혀 있으므로 반드시 기능 테스트 후 진행
- `PlayerManager` / `MonsterManager`는 싱글톤 인스턴스 — import 경로 바뀌면 인스턴스가 분리될 수 있으므로 주의
- `player/routers/router.py`의 `_send_visible_monsters` 헬퍼는 router와 강하게 결합 — service 이동 전 의존성 정리 필요
- Phase 1~2는 언제든 안전하게 진행 가능
- Phase 3~5는 한 번에 하나씩, 도메인 단위로 진행 권장
