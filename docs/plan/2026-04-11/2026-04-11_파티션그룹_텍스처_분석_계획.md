# 파티션 그룹 텍스처 시스템 분석 및 구현 계획

**작성일**: 2026-04-11  
**목표**: world_partition_group 기반으로 인접 파티션을 재그룹핑하고, 그룹 단위 페르소나 통일 바닥 텍스처 적용

---

## 1. 현황 분석 리포트

### 1-1. 실제 DB 구조 (확인됨)

3개 테이블이 DB에 존재함 (migration `c8d9e0f1a2b3_add_world_partition_group_tables.py`로 생성):

```
world_partition_group          ← 그룹 메타 (페르소나, 테마, 이미지 프롬프트)
├── id
├── group_key
├── admin_area_id
├── group_seq
├── display_name
├── theme_code                 ← 그룹 공통 테마
├── persona_tag                ← 그룹 공통 페르소나
├── texture_style_profile      ← 그룹 공통 텍스처 스타일
├── image_prompt_base          ← ComfyUI 프롬프트 (그룹 공통)
├── image_prompt_negative
├── boundary_geojson           ← 그룹 전체 영역 polygon
└── is_active

world_partition_group_member   ← 파티션↔그룹 참조 테이블
├── id
├── group_id → world_partition_group.id
├── level_id → world_level_partition.id
├── member_order
├── member_role (road/level)
└── weight (area_m2 기반)

world_level_partition          ← 개별 파티션 (기존)
├── group_key (레거시, 그룹 이동 후에도 유지)
├── group_theme_code (레거시)
└── persona_tag (파티션 레벨 세부 태그)
```

**마이그레이션 시점에 데이터 이전 완료**: `world_level_partition.group_key` 기준으로 그룹이 자동 생성되고 member 테이블에 매핑됨.

## 2. 목표 상태 (이미지 2, 3 참고)

현재 상태(이미지 1): 파티션이 단색 또는 단조로운 바닥 텍스처로 표시됨

목표 상태(이미지 2, 3):
- 인접한 파티션들이 그룹으로 묶여 **동일한 페르소나의 바닥 텍스처** 공유
- 그룹 경계선이 시각적으로 명확히 구분됨
- 그룹 내부 파티션들은 같은 테마/스타일로 통일감 있게 표시
- 각 그룹은 고유한 배경 텍스처(잔디, 도시블록, 학교운동장 등)로 차별화

## 3. ComfyUI 오프라인 텍스처 생성 방식 평가

**방식**: `world_partition_group.boundary_geojson` polygon 꼭지점 + `image_prompt_base` → ComfyUI → PNG 텍스처 파일 → `/public/ground/`에 배치 (오프라인 사전 생성)

**적합성 판단: 적합 (오프라인 파이프라인 전제)**

| 항목 | 평가 |
|---|---|
| 그룹 경계 형태 반영 | polygon 마스크 활용 → 경계가 지형과 일치 |
| 페르소나 통일성 | group 레벨 image_prompt_base로 그룹 내 일관성 확보 |
| 런타임 부하 | 없음 (오프라인 사전 생성, 정적 파일 서빙) |
| 이음매 처리 | 그룹 단위로 한 장 생성 후 파티션별 마스킹이면 이음매 없음 |
| 구현 복잡도 | 중간 — Python 스크립트 + ComfyUI API 연동 필요 |

## 4. 구현 체크리스트

### Phase 0 — 모델 및 API 정비 (즉시)

- [x] `models.py`에 `WorldPartitionGroup`, `WorldPartitionGroupMember` 모델 추가
- [ ] `partition_service.py`에 그룹 조회 함수 추가
  - `get_group_by_key(group_key)` → WorldPartitionGroup
  - `get_group_members(group_id)` → List[WorldLevelPartition]
- [ ] 라우터에 그룹 정보 API 엔드포인트 추가 (프론트에서 그룹별 텍스처 로드용)
  - `GET /world/groups?admin_area_id=...` → 그룹 목록 + image_prompt_base
  - `GET /world/groups/{group_key}/members` → 해당 그룹 파티션 목록

### Phase 1 — 파티션 재그룹핑 (그룹 경계 재배치)

- [ ] 현재 `world_partition_group` 데이터 확인 (어떤 파티션이 어떤 그룹에 속하는지)
  ```sql
  SELECT g.group_key, g.display_name, g.theme_code, COUNT(m.id) as member_count
  FROM world_partition_group g
  JOIN world_partition_group_member m ON m.group_id = g.id
  GROUP BY g.id ORDER BY g.group_seq;
  ```
- [ ] 인접성 기반 그룹 재배치 기준 정의
  - 인접한 파티션(공유 엣지 있음)을 같은 그룹으로 묶기
  - `dominant_landuse`가 같은 파티션 우선 그룹핑
  - `theme_code`가 같은 파티션 우선 그룹핑
- [ ] 재그룹핑 스크립트 작성 (`back/scripts/regroup_partitions.py`)
  - 기존 `world_partition_group_member` 행 초기화
  - 인접성 + landuse 기준으로 새 group_key 할당
  - `world_partition_group` 업데이트 (display_name, persona_tag, image_prompt_base)
  - `world_level_partition.group_key` 동기화 (레거시 컬럼)
- [ ] 재그룹핑 후 데이터 검증
  ```sql
  -- 고아 파티션 확인 (그룹 없음)
  SELECT COUNT(*) FROM world_level_partition p
  WHERE NOT EXISTS (SELECT 1 FROM world_partition_group_member m WHERE m.level_id = p.id);
  ```

### Phase 2 — ComfyUI 오프라인 텍스처 생성 파이프라인

- [ ] `tools/generate_group_textures.py` 스크립트 작성
  - DB에서 `world_partition_group` 전체 로드
  - 각 그룹의 `boundary_geojson` polygon → 픽셀 좌표 변환 (lat/lon → 텍스처 UV)
  - `image_prompt_base` + `persona_tag` + `theme_code` 조합으로 ComfyUI 프롬프트 구성
  - ComfyUI API 호출 (workflow: ControlNet mask + txt2img)
  - 결과 PNG → `/front/public/ground/{group_key}.png` 저장
- [ ] ComfyUI 워크플로우 파일 작성 (`tools/workflows/group_texture_workflow.json`)
  - 입력: polygon mask 이미지 + 텍스트 프롬프트
  - 출력: 1024×1024 텍스처 PNG
- [ ] 파일명 규칙 정의 및 `CityBlockOverlay.jsx` 로딩 경로 연동
  - 현재: `hash(group_key|group_theme_code|texture_profile|partition_seq)`
  - 변경: `group_key` 직접 사용 → `/public/ground/{group_key}.png`

### Phase 3 — 프론트엔드 텍스처 렌더링 연동

- [ ] `CityBlockOverlay.jsx` 텍스처 로딩 로직 수정

## 5. 권장 실행 순서

```
Phase 0 (모델/API) → Phase 1 (재그룹핑) → Phase 4 (경계 레이어 검증)
→ Phase 2 (ComfyUI 텍스처 생성) → Phase 3 (프론트 연동) → Phase 5 (QA)
```

Phase 2 (ComfyUI)는 Phase 3 전에 완료되어야 함. 텍스처 파일이 없으면 fallback 색상으로 Phase 3 먼저 연동 테스트 가능.
