# 바닥 텍스처 그룹 단위 렌더링 계획

## 배경 및 목표

### 현재 문제
- 노량진1동 기준 `world_partition` 230개 → 바닥 메시가 230조각
- 각 파티션은 도로로 잘린 작은 블록(평균 ~8,000m²)이라 바닥이 얼기설기 쪼개져 보임
- `world_partition_group`는 이미 그룹 경계(`boundary_geojson`) + 테마(`theme_code`) 보유

### 목표
- 바닥 텍스처 타일 단위를 **그룹 수준**으로 올려 시각적으로 자연스러운 바닥 구현
- micro 파티션(위치 판정, 스폰, NPC 영역 등 게임플레이 로직)은 **그대로 유지**

---

## 분석

### 데이터 구조
```
world_partition_group          ← 그룹 경계 + 테마 (이미 있음)
  └─ world_partition_group_member
       └─ world_partition      ← 미세 파티션 (게임플레이 용)
```

노량진1동 기준:
- 그룹 21개 (ancient_stone_route 포함), 바닥용 그룹 ~14개
- 그룹당 면적: 59,000m² ~ 200,000m²
- 한 그룹당 파티션 수: 6~27개

### 렌더링 분리 원칙

| 역할 | 데이터 소스 | 단위 |
|------|------------|------|
| 바닥 텍스처 메시 | `world_partition_group` | 그룹 1개 = 바닥 타일 1~3개 |
| 현재 위치 판정 | `world_partition` | micro 파티션 |
| NPC/스폰 배치 | `world_partition` | micro 파티션 |
| 파티션 경계선 오버레이 | `world_partition` | micro 파티션 |

### 그룹 boundary 크기 분류 (노량진1동)

| 크기 | 분할 수 | 해당 그룹 수 |
|------|---------|------------|
| < 50,000m² | 1개 | ~5개 |
| 50,000 ~ 150,000m² | 2개 | ~10개 |
| > 150,000m² | 3개 (longest_axis) | ~6개 |

## 구현 방향: 두 가지 선택지

### 안 A — 프론트 전용 변경 (권장)
그룹 boundary_geojson을 바닥 렌더링에 사용. DB 스키마 변경 없음.

1. `worldApi.getDongGroups(dongId)` 엔드포인트 추가
   - `world_partition_group` 쿼리 (members ≥ 1인 것만)
   - 반환: `group_key, group_seq, display_name, theme_code, boundary_geojson, total_area_m2`

2. `CityBlockOverlay.jsx` (또는 새 `GroupFloorOverlay.jsx`)
   - 그룹 boundary_geojson → ShapeGeometry → 텍스처 UV 매핑
   - 큰 그룹(> 150,000m²)은 `longest_axis_split` 방향으로 2~3분할해서 텍스처 타일링

3. 기존 partition 기반 바닥 렌더링 비활성화 (또는 불투명도 0으로)

### 안 B — DB world_partition 재생성
micro 파티션 자체를 그룹 단위로 재생성. 그룹 1개 = 파티션 1~3개.

- 장점: DB가 렌더링 단위와 일치
- 단점: 게임플레이용 micro 단위 영구 손실, 재생성 비용 큼
- **권장 안 아님** — 나중에 micro 단위 필요할 때 다시 쪼개야 함

### 안 C — world_partition_floor_tile 별도 테이블 추가 (미래용)
텍스처/floor 전용 폴리곤을 별도 테이블로 관리.

```sql
CREATE TABLE world_partition_floor_tile (
  id SERIAL PRIMARY KEY,
  group_id INT REFERENCES world_partition_group(id),
  floor_seq INT,           -- 그룹 내 분할 순서
  boundary_geojson JSON,
  theme_code TEXT,
  texture_image_url TEXT
);
```
- 장점: 렌더링 단위와 게임플레이 단위 완전 분리
- 단점: 테이블 추가 및 생성 스크립트 필요

## 주의사항

- `world_partition` micro 파티션 **절대 삭제 금지** — 게임플레이 로직 의존
- 그룹 boundary는 `regroup_partition_members.py`가 계산하는 convex hull 기반 → 정확한 외곽이 아닐 수 있음. 실제 파티션 union이 필요하다면 별도 계산 필요
- 렌더링 레이어 순서: 그룹 바닥(최하단) → 파티션 경계선 오버레이 → NPC/오브젝트
