# 성능 병목 리팩토링 분석
> 작성일: 2026-04-12  
> 요약: 파티션/몬스터 시스템 전반의 불필요한 반복 계산 및 폴링 패턴 정리

---

## 배경 및 목표

플레이어 위치 기반으로 파티션 텍스처와 몬스터를 주변에만 로딩하는 시스템이 구현되어 있다.  
그러나 실제로는 플레이어가 정체해 있거나 동일 데이터를 반복 요청하는 상황에서도  
백엔드·프론트 모두에서 계속 계산이 발생하고 있어 렉/병목의 원인이 된다.

**목표**: 불필요한 계산을 제거하고, 캐싱·공간 인덱싱·폴링 최적화로 성능을 개선한다.

---

## 우선순위별 할 일 체크리스트

### Phase 1 - 즉각적 효과, 위험도 낮음 (1~2일)

- [ ] **A: 지역 감지 인터벌 통합**
  - `useCurrentRegionInfo.js`와 `RpgWorld.jsx`의 두 인터벌을 하나로 병합
  - 플레이어가 N미터 이상 이동했을 때만 갱신 (이벤트 기반으로 전환 검토)
  - 통합 후 불필요한 `setCurrentDistrictId` 중복 제거

- [ ] **D1/D2: RpgWorld visibleMonsterElements useMemo 수정**
  - `debugConfig.playerScale`을 의존성에서 제거 (몬스터 컬링과 무관)
  - 플레이어 위치 기반 컬링 로직을 `useFrame` 루프 또는 `monsters` 변경 시점에 실행

- [ ] **C1/C2: 파티션 데이터 공유 (Context 또는 상위 상태)**
  - `PartitionBoundaryOverlay`와 `GroupColorOverlay`가 동일 API를 공유하도록
  - 상위 컴포넌트에서 한 번 fetch → props 또는 Context로 전달
  - 동 ID가 같으면 재요청 금지 (간단한 메모이제이션)

- [ ] **C3: findAdjacentGroupKeys 캐싱**
  - `(currentGroupKey, partitions 해시)` 기준으로 결과 메모이제이션
  - `useMemo` 의존성을 `partitions` 길이 + `currentGroupKey` 조합으로 단순화

---

### Phase 2 - 백엔드 핵심 계산 최적화 (2~3일)

- [ ] **B1: MonsterManager 게임루프 공간 인덱싱 도입**
  - 현재: 모든 몬스터 × 모든 플레이어 O(n×m) 브루트포스
  - 개선: 그리드 버킷(Cell Bucket) 방식으로 근처 엔티티만 탐색
    - 25m DETECT_RANGE 기준으로 셀 크기 30m 그리드 분할
    - 몬스터/플레이어 이동 시 버킷 업데이트
    - 인접 9개 셀만 조회 → 평균 탐색 대상 대폭 감소
  - sqrt 제거: `distSq < DETECT_RANGE ** 2` 비교로 대체

- [ ] **B2: 몬스터 tier 카운트 캐싱**
  - `_normal_count` 변수를 클래스 멤버로 유지
  - 스폰/제거 시점에만 업데이트 (게임루프에서 매번 sum 불필요)

- [ ] **B3: _send_visible_monsters 쓰로틀링 강화**
  - 현재: 15m 이동 OR 2초 중 하나 → 너무 자주 트리거
  - 개선: 플레이어당 독립 타이머, 이동 거리 임계값 상향 (예: 25m)
  - 몬스터 총 개수 변화 없으면 전송 스킵

- [ ] **F1: 파티션 위치 판정 캐싱**
  - `(dong_osm_id, lat_rounded, lng_rounded)` 조합으로 LRU 캐시 적용
  - Shapely 반복 파싱 제거: `shape()` 결과를 파티션 로드 시점에 미리 변환하여 저장

### Phase 3 - 자동전투 최적화 (1일)

- [ ] **E1: useAutoFarm 이벤트 기반 재탐색**

  - 타겟 변경 또는 이동 이벤트로 재시작

## 주의사항

### 건드리면 안 되는 영역
- **파티션 경계 GeoJSON 데이터** - 백엔드 DB 소스 변경 금지
- **몬스터 스폰 규칙** (`agents/game_design/`) - 게임 밸런스 영향 금지
- **WebSocket 프로토콜 메시지 타입** - 양측 동시 변경 필요, 단독 변경 금지

### 검증 방법
1. Phase 1 완료 후: 브라우저 Network 탭에서 `/region_info` API 호출 빈도 확인
2. Phase 2 완료 후: 백엔드 로그에서 game_loop 실행 시간 측정
3. 전체 완료 후: 플레이어 5분 이상 정체 시 CPU/메모리 사용량 비교

### 리스크
- **MEDIUM**: 지역 감지 통합 시 `worldLoadStage` 리셋 타이밍이 달라질 수 있음 → 텍스처 로딩 테스트 필수
- **MEDIUM**: 공간 인덱싱 도입 시 버킷 경계에서 몬스터가 감지 안 될 수 있음 → DETECT_RANGE보다 셀 크기를 충분히 크게 설정
- **LOW**: useMemo 의존성 변경 시 일시적 렌더링 누락 가능 → 변경 후 즉시 시각 확인

## 요약

렉의 주요 원인은 3가지다:

1. **중복 폴링**: 두 개의 지역 감지 인터벌이 동시 실행 중 → 통합
2. **O(n×m) 거리 계산**: 백엔드 0.1s 루프에서 공간 인덱싱 없이 브루트포스 → 그리드 버킷
3. **캐싱 없는 중복 API**: 두 컴포넌트가 동일 데이터 각자 요청, 매번 Shapely 재계산 → 공유 + 캐시

Phase 1만 완료해도 프론트 쪽 불필요한 계산 70% 이상 제거 가능하다.  
Phase 2는 플레이어/몬스터 수가 늘어날수록 효과가 커진다.
