## 결정 배경

### 현재 방식의 문제

| 항목 | 현황 |
|------|------|
| 노량진1동 파티션 수 | 268개 |
| 서울 전체 법정동 | 424개 |
| 필요한 고유 텍스처 수 | 최대 11만 장 이상 |
| 파티션 경계 문제 | 폴리곤마다 UV가 달라 경계선이 눈에 띔 |
| 개발 공수 | ComfyUI 파이프라인 + 생성 + 검수 → 1인 개발로 불가 |

`agents/game_design/world_texture_terrain_direction.md` 에 이미 기록됨:
> "폴리곤별 이미지 재생성보다, 연속 지형 위에 파티션 마스크를 얹는 방향이 더 좋다"

### 결정

- **지금**: theme_code 기반 5~6종 텍스처 타일링 (world-space UV)
- **나중**: 게임 루프 완성 후 연속 지형 + 파티션 마스크 방식으로 업그레이드
- **영구 폐기**: 파티션/그룹별 고유 이미지 한 장씩 생성하는 방식

|------------|------------|------|

> 새로 생성 불필요 — 기존 보유 텍스처 재활용 우선.

## 핵심 기술 방향: world-space UV 타일링

파티션 폴리곤 경계와 무관하게 **월드 좌표 기준으로 UV를 계산**한다.  
인접 파티션이 같은 theme이면 텍스처가 끊김 없이 이어진다.

```glsl
/* ShaderMaterial 또는 MeshStandardMaterial onBeforeCompile */
/* fragment: world position x/z → UV 계산 */
vec2 worldUV = vWorldPosition.xz / TILE_SIZE;
vec4 color = texture2D(uTexture, fract(worldUV));
```

또는 Three.js에서 간단하게:
```js
// ShapeGeometry 생성 후 UV를 world position 기준으로 덮어씀
geometry.attributes.uv 를 position.x/z 로 재계산
```

## 구현 범위

### 변경할 것

- `CityBlockOverlay.jsx` — UV 계산 방식을 world-space 기반으로 변경
- `GroupFloorOverlay.jsx` (신규 예정이면 처음부터 world-space UV로 작성)
- theme_code → 텍스처 파일 매핑 상수 추가

### 건드리지 않을 것

- `world_partition` 테이블 — 게임플레이 로직 의존, 절대 변경 없음
- `world_partition_group` 테이블 — 그룹 경계 데이터 유지
- `SeoulTerrain.jsx` 도로 렌더링 — 별도 계층, 이번 범위 아님
- ComfyUI 파이프라인 스크립트 — 삭제하지 말고 보관 (추후 특수 지형용 가능)

## 우선순위 판단 근거

모바일 자동파밍 RPG 유저의 시선:

```
전투/스킬 이펙트       ← 항상 봄
HP바/데미지 숫자       ← 항상 봄
UI (인벤, 도감, 퀘스트) ← 자주 봄
몬스터 외형            ← 자주 봄
바닥 텍스처            ← 거의 안 봄 (카메라 각도상 작게 보임)
```

바닥 텍스처 퀄리티가 유저 리텐션에 미치는 영향: 낮음  
도감 수집 재미, 펫 뽑기 연출, 레벨업 체감이 리텐션을 만든다.

→ **Phase 1만 해도 충분. Phase 2~3는 게임 루프 완성 후.**
