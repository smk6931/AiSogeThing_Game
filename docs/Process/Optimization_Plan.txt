이슈 분석 및 최적화 진행 방향 보고서 (2026-03-09)

1. 현재 주요 부하 요소 (Performance Bottlenecks)

   - 지형 데이터 기반 정점 페인팅 (SeoulHeightMap.jsx)
     - 원인: 모든 지형 정점(Grid)에 대해 실시간으로 GPS 역산을 수행하고, 
       수백 개의 폴리곤을 대상으로 충돌 검사(PiP)를 실시함.
     - 영향: 구 전환 시 텍스처 업데이트 과정에서 심한 프리징 발생.
   
   - 다중 레이어 중첩 (Overdraw)
     - 원인: 바닥 지도, 구역 오버레이, 시티 블록, 도로 지형, 등고선이 중첩됨.
     - 영향: GPU 렌더링 부하 가중 및 투명도 연산 증가.

   - 메인 스레드 지오메트리 생성
     - 원인: 도로 및 구역의 수천 개 메시를 CPU에서 하나로 합치고 높이를 계산함. (구가 너무 커서 데이터 처리량이 과도함)
     - 영향: 데이터 로딩 완료 시점에 순간적인 프레임 드랍 및 메모리 점유율 상승.

2. 데이터 분할 전략 고도화 (Tiling Strategy)

   - 현행: 구(District, admin_level 6) 단위 로드
   - 개선안 A (행정 구역 세분화): 동(Dong, admin_level 8) 단위 로드
     - 장점: 실제 지명 기반으로 유저 인지가 쉬움.
     - 단점: 동별 크기가 편차가 심하고 경계면 정합(Stitching)이 복잡함.
   - 개선안 B (기술적 그리드): 일정 크기(예: 500m x 500m) 그리드 단위 로드
     - 장점: 데이터 크기가 일정하여 로딩 성능 예측이 가능하고 관리하기 쉬움.
     - 단점: 유저에게 의미 있는 단위(동네 이름 등)를 제공하려면 추가 처리가 필요함.

   => 전략: 우선 '동(Dong)' 단위 추출 API를 구축하여 데이터 사이즈를 줄이되, 내부적으로는 쿼드트리(Quadtree)나 그리드 형태의 스트리밍을 병행 검토.

3. 등고선 및 지형 시스템 개선 방향 (진행 중)

   - 등고선 형태 변경 (계단식 구현)
     - 방법: 고도 값(Normalization)을 특정 단위로 양자화(Rounding) 처리.
     - 기대효과: 부드러운 곡면 대신 강렬한 계단 형태의 맵 디자인 가능 (최적화와 디자인 동시 확보).

   - 텍스처 단순화
     - 방법: 연산량이 많은 3중 텍스처 블렌딩을 일시 중단하고 플랫한 컬러(Green계열) 위주로 변경.
     - 기대효과: GPU 부하 감소 및 시인성 확보.

   - 연산 최적화
     - 방법 1: CPU 기반 PiP 연산을 중단하고, 미리 생성된 구역별 마스크 이미지를 쉐이더에서 참조.
     - 방법 2: Web Worker를 사용하여 메인 스레드 멈춤 현상 제거.

3. 단계별 실행 계획

   - 1단계 (즉시 실행 - 완료): 
     1. SeoulHeightMap.jsx의 쉐이더 텍스처 제거 및 단색(초록색) 변경.
     2. 지형 높이 계산 로직 양자화 (10m 계단식 지형).
     3. 부하가 큰 Zone 기반 정점 페인팅 로직 중단.
   
   - 2단계 (데이터 세분화 - 진행 예정):
     1. 백엔드: 동(Dong) 단위 행정 경계 패치 및 지형 추출 API 개발.
     2. 프론트엔드: 현재 플레이어 좌표 기준 '동'을 판별하고 주변 '동' 데이터를 스트리밍 로드.
     3. 지형 데이터를 타일 단위로 쪼개서 관리하는 TerrainStreamingSystem 활성화.

   - 3단계 (성능 고도화):
     1. Web Worker를 사용하여 지오메트리 생성 연산을 메인 스레드와 분리.
     2. 레이어 통합을 통한 Draw Call 및 Overdraw 최소화.
