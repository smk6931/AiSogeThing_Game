# 파티션 그룹핑 개념 정리

## 핵심

`world_level_partition`는 계속 **하나의 테이블**로 관리한다.

하지만 이 테이블의 **한 row는 끝까지 "가장 작은 파티션 1개"** 를 뜻해야 한다.

즉:

- row 1 = 작은 파티션 1개
- row 2 = 작은 파티션 1개
- row 3 = 작은 파티션 1개

이 규칙은 유지한다.

대신 각 row에 아래 같은 **그룹용 칼럼**을 추가한다.

- `group_key`
- `group_seq`
- `group_display_name`
- `group_theme_code`

## 지금 구조

현재는 대충 이런 느낌이다.

```text
world_level_partition

row 1 = 작은 파티션 A
row 2 = 작은 파티션 B
row 3 = 작은 파티션 C
row 4 = 작은 파티션 D
...
```

즉 UI, 경계선, 이름이 전부 작은 파티션 기준으로만 움직이기 쉽다.

그러면 문제가 생긴다.

- 이름이 너무 자주 바뀜
- 파티션 수가 너무 많아 보임
- 플레이어가 체감하는 "지역"이 너무 잘게 쪼개짐

## 바꿀 구조

한 테이블은 그대로 둔다.

대신 같은 테이블 row마다 "이 파티션이 어느 그룹에 속하는지"를 기록한다.

```text
world_level_partition

row 1 = 작은 파티션 A, group_key = G01
row 2 = 작은 파티션 B, group_key = G01
row 3 = 작은 파티션 C, group_key = G01
row 4 = 작은 파티션 D, group_key = G02
row 5 = 작은 파티션 E, group_key = G02
row 6 = 작은 파티션 F, group_key = G03
...
```

즉:

- row는 여전히 작은 파티션
- 여러 row가 같은 그룹 이름을 공유할 수 있음

## 그림으로 보면

현재 micro partition:

```text
+----+----+----+----+
| A1 | A2 | B1 | B2 |
+----+----+----+----+
| A3 | A4 | C1 | C2 |
+----+----+----+----+
| D1 | D2 | C3 | C4 |
+----+----+----+----+
```

이걸 그룹으로 묶으면:

```text
+---------+---------+
|   G01   |   G02   |
| A1 A2   | B1 B2   |
| A3 A4   | C1 C2   |
+---------+---------+
|   G03   |   G02   |
| D1 D2   | C3 C4   |
+---------+---------+
```

여기서 뜻은 이렇다.

- `A1`, `A2`, `A3`, `A4`는 작은 파티션
- 하지만 전부 `G01` 그룹에 속함
- 그래서 UI에서는 `G01`의 이름 하나만 보여줄 수 있음

## 왜 `type=sector/group`으로 섞으면 안 좋나

헷갈리기 쉬운 방식:

```text
row 1 = sector
row 2 = sector
row 3 = sector
row 4 = group
row 5 = group
```

이렇게 되면 한 테이블 안에

- 작은 파티션 row
- 큰 그룹 row

가 같이 섞여 들어간다.

그러면 나중에 아래 로직이 더러워진다.

- 현재 위치 판정
- 경계선 표시
- 텍스처 덮기
- UI 지역명 출력
- 인접 구역 계산

즉 같은 테이블이라도
"서로 다른 크기의 geometry row를 같이 넣는 것"은 피하는 게 좋다.

## 내가 제안한 방식

한 테이블 유지:

```text
world_level_partition
```

한 row 의미 유지:

```text
1 row = 가장 작은 파티션 1개
```

추가되는 것:

```text
group_key
group_seq
group_display_name
group_theme_code
```

즉 같은 테이블 안에서
"작은 파티션이 어느 그룹에 속하는가"만 관리한다.

## 실제 게임에서는 어떻게 쓰나

### 1. 현재 위치 판정

플레이어는 여전히 가장 작은 파티션 기준으로 판정한다.

```text
player -> p042 안에 들어감
```

### 2. UI 지역 이름

하지만 화면에는 `p042` 자체 이름이 아니라
그 row가 속한 그룹 이름을 띄울 수 있다.

```text
p042.group_display_name = "기원의 뜰목"
```

즉 실제 내부 판정은 작은 셀 기준,
플레이어에게 보이는 이름은 그룹 기준이 된다.

### 3. 경계선

경계선도 2단계로 나눌 수 있다.

```text
micro 경계 = 모든 작은 파티션 경계
group 경계 = group이 달라지는 바깥 경계만 표시
```

그래서 나중에는 옵션으로 나눌 수 있다.

- 디버그 모드: micro 경계
- 실제 플레이 UI: group 경계

### 4. 텍스처

텍스처도 선택지가 2개다.

```text
micro texture = 작은 파티션마다 다른 텍스처
group texture = 같은 group에 속한 파티션은 비슷한 무드로 통일
```

추천은 이렇다.

- 판정: micro 기준
- 지역명: group 기준
- 큰 무드/텍스처: group 기준
- 디버그 확인: micro 기준

## 한 줄 요약

네가 말한:

```text
한 테이블에 칼럼 넣어서 관리
```

이 방향 자체는 맞다.

하지만 중요한 건 이것이다.

```text
group을 "새 row 타입"으로 넣지 말고
기존 micro partition row에 "그룹 소속 정보" 칼럼을 넣자
```

그게 나중에 제일 안 꼬인다.
