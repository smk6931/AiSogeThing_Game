# 바이브코딩 에이전트 운영 감사

작성일: 2026-04-05
대상 프로젝트: `C:\GitHub\AiSogeThing_Game`
분석 범위:
- 프로젝트 로컬: `AGENTS.md`, `agents/`, `CLAUDE.md`, `CODEX.md`, `.agent/skills`
- 사용자 글로벌: `C:\Users\addmin\.claude\*`, `C:\Users\addmin\.codex\*`

## 1. 한줄 결론

지금의 작업 방식은 "에이전트를 안 쓰는 사람"이 아니라 "많이 쓰는 사람"에 가깝다. 다만 잘 쓰는 축과 엉뚱하게 쓰는 축이 동시에 있다.

- 잘 쓰는 축:
  - 규칙을 문서화했다.
  - 프로젝트별 지침을 따로 뒀다.
  - 계획, 검증, 리뷰 같은 메타 작업을 커맨드/에이전트로 분리했다.
  - Claude와 Codex를 둘 다 실험하면서 자기 작업 방식에 맞게 커스터마이징했다.

- 엉뚱하게 쓰는 축:
  - 규칙이 너무 많은 레이어에 흩어져 있다.
  - 같은 내용을 여러 번 적고 있다.
  - 일부 규칙은 서로 철학이 다르다.
  - 권한 자동화가 너무 넓게 열려 있는 부분이 있다.
  - "문서가 많음 = 운영이 정교함" 상태가 아니라 "문서가 많아서 에이전트가 매번 다른 소스에 끌려갈 수 있는" 상태다.

즉, 감으로 잘 쓰고는 있지만, 지금 상태로 규모가 커지면 생산성보다 컨텍스트 충돌 비용이 더 커질 가능성이 높다.

## 2. 현재 구조 진단

### 2.1 규칙 소스가 5계층이다

현재 실제로 에이전트 행동에 영향을 주는 규칙 소스는 대략 아래 5개 층이다.

1. 프로젝트 루트 `AGENTS.md`
2. 프로젝트 `agents/` 문서군
3. 프로젝트 루트 `CLAUDE.md`, `CODEX.md`
4. 프로젝트 `.agent/skills/*`
5. 사용자 홈 `C:\Users\addmin\.claude\*`, `C:\Users\addmin\.codex\*`

이 구조의 문제는 "규칙이 없다"가 아니라 "규칙 우선순위가 사람 머릿속에만 있다"는 점이다.

### 2.2 좋은 점

- `agents/README.md`와 `AGENTS.md`가 "필요한 문서만 읽어라"는 원칙을 갖고 있다.
- `agents/`가 도메인별로 나뉘어 있어, 최소 컨텍스트 전략 자체는 좋다.
- `CLAUDE.md`, `CODEX.md`가 프로젝트 공통 규칙 진입점 역할을 하고 있다.
- `.claude/commands/plan.md`, `.claude/commands/verify.md`, `.claude/commands/code-review.md`처럼 반복 작업을 템플릿화해 둔 점은 좋다.
- `.agent/skills/agent-guidelines/SKILL.md`가 "예스맨 금지"를 강하게 박아둔 점은 매우 좋다.

### 2.3 구조적 문제

#### 문제 A. 같은 규칙이 여러 곳에 중복된다

예:
- `AGENTS.md`와 프로젝트 `CODEX.md`, `CLAUDE.md`가 모두 `agents/` 우선 참조를 말한다.
- 프로젝트 `.agent/skills/communication-standards`와 홈 `.claude/CLAUDE.md`가 모두 커뮤니케이션 방식에 개입한다.
- 홈 `.codex/README.md`와 실제 `.codex/rules/default.rules`의 설명 철학이 다르다.

중복 자체가 악은 아니다. 하지만 지금은 "요약-상세" 구조가 아니라 "비슷한 규칙이 다른 말투로 여기저기 있음"에 가깝다.

#### 문제 B. 일부 규칙은 툴 철학과 충돌한다

가장 큰 예:

- `C:\Users\addmin\.codex\README.md`는 좁은 prefix만 허용하라고 설명한다.
- 그런데 실제 `C:\Users\addmin\.codex\rules\default.rules`에는 아래 규칙이 있다.

`prefix_rule(pattern=["C:\\WINDOWS\\System32\\WindowsPowerShell\\v1.0\\powershell.exe", "-Command"], decision="allow")`

이건 사실상 PowerShell `-Command` 전체 허용에 매우 가깝다.
즉, 문서상 철학은 "좁게 허용"인데 실제 운영은 "꽤 넓게 허용"이다.

이건 단순 보안 이슈만이 아니라, 작업 로그 품질에도 영향을 준다.
에이전트가 무엇을 승인 없이 해도 되는지 경계가 흐려지면, 작업 요청 문장도 느슨해지고 검증 습관도 약해진다.

#### 문제 C. Claude와 Codex의 역할 분리가 문서상 명확하지 않다

현재 보이는 성향은 이렇다.

- Claude:
  - 커맨드 체계가 풍부함
  - planner/architect/reviewer/debugger 같은 역할 에이전트가 명시적임
  - `bypassPermissions` 성향으로 빠르게 밀어붙이기 좋음

- Codex:
  - 프로젝트 신뢰도와 prefix rule 중심
  - skills와 workspace 규칙을 더 세밀하게 읽음
  - 상대적으로 절차적이고 제약을 잘 따름

문제는 이 차이가 "역할 분담"으로 정리된 문서가 없다는 점이다.
그래서 실제 사용 시:

- 어느 작업을 Claude에 던질지
- 어느 작업을 Codex에 던질지
- 둘 다 같은 문서를 수정하게 만들지

가 세션마다 달라질 수 있다.

#### 문제 D. `.agent/skills`와 `agents/`가 역할이 겹친다

둘 다 "에이전트가 참고할 작업 지침"인데 성격 구분이 흐리다.

- `agents/`: 프로젝트 규칙, 도메인 설계, 작업 계획
- `.agent/skills/`: 코딩 스타일, 구조 스타일, 소통 방식, 터미널 실행 방식

이 자체는 가능하다. 그러나 현재는 사용자 입장에서:
"이 규칙을 agents로 둘지, .agent skill로 둘지" 선택 기준이 약하다.

그 결과, 새 규칙이 생길 때마다 가장 먼저 떠오르는 아무 레이어에 추가될 위험이 있다.

## 3. 당신이 이미 잘하고 있는 것

### 3.1 "무규칙 바이브코딩"은 아니다

많은 사람이 바이브코딩을 한다고 말하지만 실제로는 매번 즉흥 프롬프트만 던진다.
당신은 그 수준은 이미 넘었다.

지금 구조를 보면 이미 아래를 하고 있다.

- 프로젝트 전용 룰 분리
- 작업 전 읽을 문서 인덱싱
- 계획 명령과 검증 명령 분리
- 아키텍처 철학 문서화
- 코드리뷰 기준 문서화
- 서브에이전트 역할 분리 시도

이건 "AI를 IDE 자동완성처럼만 쓰는 단계"보다 훨씬 앞이다.

### 3.2 "사고 외주화"를 어느 정도 시스템화했다

특히 좋은 포인트:

- planner를 따로 둠
- verify를 따로 둠
- code-review를 따로 둠
- agents 문서를 도메인별로 쪼갬

이건 "생성"만 시키는 게 아니라 "생성 전/후 사고 단계"도 분리한 것이다.
바이브코딩에서 제일 중요한 건 생성보다 검증 루프인데, 그 방향은 맞다.

### 3.3 초보 사용자용 안전장치를 일부 잘 걸어뒀다

`.agent/skills/agent-guidelines/SKILL.md`의 핵심은 매우 좋다.

- 예스맨 금지
- 장단점 비교 강제
- 실무 관점 설명
- 최종 선택권은 사용자에게

이건 에이전트를 "비서"가 아니라 "기술 고문"으로 쓰려는 방향이다.
이 철학은 유지하는 것이 맞다.

## 4. 지금 엉뚱하게 쓰고 있는 부분

### 4.1 문서를 너무 많이 두고, 진짜 소스 오브 트루스는 적지 않았다

문서가 많아도 괜찮다. 문제는 역할 구분이다.

지금은 다음 질문에 즉답하기 어렵다.

- 프로젝트 공통 규칙의 최상위 소스는 어디인가?
- Claude 전용 규칙은 어디까지인가?
- Codex 전용 규칙은 어디까지인가?
- 스타일 규칙은 어디까지 강제인가?
- 설계 원칙과 개인 취향은 어떻게 구분되는가?

현재 상태는 "문서가 많아서 정교함"이 아니라 "문서가 많아서 해석 비용이 큼" 쪽에 가깝다.

### 4.2 규칙의 강도 차이가 없다

예를 들어 아래는 같은 무게로 보이면 안 된다.

- 보안/민감정보 규칙
- DB source of truth 규칙
- import 스타일 취향
- 한국어 보고 스타일

하지만 현재 문서 구조에서는 이런 것들이 비슷한 레벨로 섞여 있다.
그 결과 에이전트가 본질적 규칙과 취향 규칙을 동일한 우선순위로 다룰 수 있다.

추천 분류는 아래다.

- Level 1: 절대 규칙
  - 보안, 데이터 무결성, destructive command, source of truth
- Level 2: 아키텍처 규칙
  - 레이어 경계, 폴더 구조, API 위치
- Level 3: 작업 프로세스 규칙
  - plan, verify, review, 보고 방식
- Level 4: 스타일 선호
  - import 스타일, 변수명 스타일, 말투 선호

### 4.3 OOP/Singleton/Namespace import 취향이 너무 전역 규칙처럼 올라와 있다

`.agent/skills/coding-standards/SKILL.md`는 강한 취향 문서다.

- `import * as ModuleName` 선호
- 주요 로직은 반드시 클래스
- 매니저는 싱글톤
- Unreal 스타일 강제

이건 프로젝트에 따라 매우 유용할 수도 있지만, "항상 맞는 일반 규칙"은 아니다.
특히 JS/React 영역에서는 함수형 패턴과 훅이 더 자연스러운 경우가 많다.

즉, 이 문서는 "전역 코딩 철학"이 아니라 "이 프로젝트의 특정 영역에서만 쓰는 패턴"으로 내려야 한다.
안 그러면 에이전트가 문제에 맞는 설계보다 취향에 맞는 설계를 먼저 제안할 수 있다.

### 4.4 Claude 권한 운영은 빠르지만, 습관을 나쁘게 만들 수 있다

`C:\Users\addmin\.claude\settings.json`:

- `defaultMode = "bypassPermissions"`
- `skipDangerousModePermissionPrompt = true`

이 설정은 작업 속도는 좋다.
하지만 바이브코딩 초중기 사용자에게는 아래 부작용이 있다.

- 에이전트가 뭘 실행했는지 덜 의식하게 됨
- 명령 실행 전 확인 루프가 약해짐
- 작업 요청을 더 대충 써도 된다는 습관이 붙음

속도는 올라가지만, 장기적으로는 프롬프트 정밀도와 검증 습관이 떨어질 수 있다.

### 4.5 "명령 자동화"와 "작업 절차화"가 섞여 있다

좋은 자동화는 두 종류가 있다.

1. 실행 자동화
   - 빌드, 테스트, 서버 실행, 조회

2. 사고 자동화
   - 계획, 리뷰, 검증, 체크리스트

지금은 둘 다 많이 해놨는데, 실행 자동화보다 사고 자동화 문서가 더 약간 느슨하다.
예:

- `/verify`는 강하다.
- `/plan`도 강하다.
- 하지만 "언제 반드시 `/plan`을 타야 하고, 언제 바로 구현해도 되는가"는 프로젝트 운영 규칙으로 명확하지 않다.

즉, 커맨드는 있는데 운영 프로토콜이 덜 정리돼 있다.

## 5. 가장 중요한 핵심 문제 3개

### 5.1 진짜 핵심 문제: 규칙 계층이 너무 많다

현재는 에이전트를 잘 쓰는 게 아니라 "에이전트 규칙 생태계"를 운영하고 있다.
이건 일정 규모 이상에서는 오히려 발목을 잡는다.

정리 원칙:

- 한 규칙은 한 곳에만 둔다.
- 다른 곳에는 링크만 둔다.
- 요약 문서와 상세 문서를 분리한다.

### 5.2 진짜 핵심 문제: 권한 모델이 문서와 실제가 다르다

Codex 문서는 좁은 prefix를 권장한다.
실제 rules는 PowerShell 전체 허용이 있다.

이건 반드시 정리해야 한다.
이 상태는 "안전하게 자동화"가 아니라 "광범위 허용 후 문서로만 절제 권고"에 가깝다.

### 5.3 진짜 핵심 문제: 툴별 역할 전략이 없다

Claude와 Codex를 둘 다 쓰는 건 좋다.
문제는 둘의 역할이 명확하지 않으면 아래가 생긴다.

- 같은 문제를 두 툴에 중복 설명
- 서로 다른 문체와 규칙으로 결과물 일관성 하락
- 한쪽은 빠르게 밀고, 다른 쪽은 규칙을 더 따져서 흐름 충돌

## 6. 앞으로 더 잘 쓰는 방법

### 6.1 규칙 체계를 3층으로 줄여라

추천 구조:

1. 프로젝트 최상위 운영 헌장
   - 파일: `AGENTS.md`
   - 역할: 진입점, 우선순위, 문서 지도
   - 내용: "무엇이 source of truth인지"만 적음

2. 프로젝트 상세 규칙
   - 폴더: `agents/`
   - 역할: 도메인 규칙, 아키텍처 규칙, 작업 계획
   - 내용: 진짜 프로젝트 규칙만

3. 툴 전용 개인 설정
   - 홈 `.claude`, 홈 `.codex`
   - 역할: 툴 기능, 승인 정책, 개인 생산성 설정
   - 내용: 프로젝트 규칙 금지, 툴 운영 규칙만

반대로 아래는 줄이는 게 좋다.

- 프로젝트 루트 `CLAUDE.md`, `CODEX.md`는 최소화
- `.agent/skills`는 정말 반복되는 절차만 남김

즉:

- 프로젝트 규칙은 `AGENTS.md` + `agents/`
- 툴 규칙은 홈 디렉터리
- 스타일/개인 취향은 최소화

이렇게 나누면 된다.

### 6.2 Claude와 Codex 역할을 명확히 나눠라

추천 분리:

- Claude에 잘 맞는 일
  - 큰 기능 기획
  - 구현 순서 설계
  - 대안 비교
  - 문서 초안
  - 빠른 브레인스토밍

- Codex에 잘 맞는 일
  - 실제 코드 수정
  - 로컬 파일 탐색
  - 제한된 범위 리팩터링
  - diff 기반 점검
  - 규칙 준수형 작업

- 둘 다 쓰지 말고 직접 결정해야 하는 일
  - 제품 방향 최종 결정
  - DB source of truth 변경
  - 과한 자동 승인 설정
  - 아키텍처 철학 변경

간단히 말하면:

- Claude = 넓게 생각
- Codex = 좁게 실행

으로 두는 게 좋다.

### 6.3 "항상 먼저 시킬 질문"을 템플릿화하라

바이브코딩 효율은 모델 품질보다 요청 품질의 영향을 많이 받는다.
앞으로는 큰 작업 전에 항상 아래 틀로 던지는 게 좋다.

```md
목표:
- 이번 작업의 최종 사용자 가치는 무엇인가

범위:
- 수정 허용 파일/폴더
- 건드리면 안 되는 파일/폴더

출력 방식:
- 바로 구현할지
- 먼저 계획만 낼지
- 리뷰만 할지

검증:
- 빌드
- 테스트
- 수동 시나리오

주의:
- 반드시 읽을 agents 문서
- 피해야 할 리팩터링
```

이렇게 하면 "대충 말해도 알아서 해주겠지" 모드에서 벗어나고, 에이전트 결과 품질이 훨씬 안정된다.

### 6.4 `/plan`, `/verify`, `/code-review`를 운영 프로토콜로 승격시켜라

지금은 명령이 존재한다.
이제 필요한 건 "언제 강제냐"다.

추천 운영:

- 아래 작업은 무조건 `plan` 먼저
  - 3파일 이상 수정
  - DB/월드 구조 변경
  - 렌더/네트워크/상태관리 구조 변경
  - 새 시스템 추가

- 아래 작업은 무조건 `verify`
  - 커밋 전
  - PR 전
  - 서버/빌드 관련 변경 후

- 아래 작업은 무조건 `code-review`
  - 보안 관련
  - API/DB 변경
  - 에이전트가 대량 생성한 코드

즉, 명령을 "있는 도구"에서 "항상 타는 게이트"로 바꿔야 한다.

### 6.5 스타일 취향 문서는 약하게, 데이터 규칙 문서는 강하게

강하게 유지할 것:

- DB source of truth
- hierarchy 규칙
- vector DB 사용 금지 범위
- destructive command 정책
- 민감정보 정책

약하게 내릴 것:

- import 스타일
- 클래스 강제
- 싱글톤 강제
- 특정 언어 톤

이건 중요하다.
바이브코딩에서 생산성을 가장 많이 깎는 것은 "잘못된 자유"보다 "과한 스타일 강제"인 경우가 많다.

### 6.6 승인 정책은 다시 조여라

특히 아래는 재검토가 필요하다.

- `C:\Users\addmin\.codex\rules\default.rules`
  - `powershell.exe -Command` 전체 허용
- `C:\Users\addmin\.claude\settings.json`
  - `bypassPermissions`
  - `skipDangerousModePermissionPrompt = true`

추천:

- 조회 명령만 넓게 허용
- 쓰기 명령은 좁은 prefix만 허용
- destructive 명령은 끝까지 수동 승인
- 프로젝트별 시작 스크립트만 선택적으로 자동 허용

속도보다 중요한 것은 "안전한 반복 가능성"이다.

## 7. 실전 권장 워크플로우

### 7.1 큰 기능 추가

1. Claude에 계획만 맡긴다.
2. 결과를 보고 범위를 줄인다.
3. Codex에 구현을 맡긴다.
4. Codex 또는 Claude로 리뷰를 한 번 더 돈다.
5. 검증 명령으로 마감한다.

### 7.2 버그 수정

1. Codex에 재현 경로 + 의심 파일 범위를 준다.
2. 원인 분석만 먼저 시킨다.
3. 수정 후 verify를 돈다.
4. 회귀 위험이 크면 code-review를 추가한다.

### 7.3 설계 논쟁

1. Claude에 옵션 비교를 시킨다.
2. 장단점 문서만 받는다.
3. 최종 결정은 직접 한다.
4. 결정 후 Codex에 구현 범위를 좁혀서 넘긴다.

### 7.4 문서/운영 정리

1. Claude에 초안 구조를 뽑게 한다.
2. Codex로 실제 프로젝트 문서 위치와 충돌 여부를 점검한다.
3. 최종 문서는 프로젝트 source of truth 한 군데에만 둔다.

## 8. 당신에게 맞는 현실적 운영 원칙

당신은 "에이전트 많이 써보는 타입"이다.
그래서 필요한 건 더 많은 기능이 아니라, 더 적은 규칙과 더 명확한 게이트다.

앞으로는 아래만 기억하면 된다.

- 에이전트 수를 늘리지 말고 역할을 줄여라.
- 규칙 문서를 늘리지 말고 source of truth를 줄여라.
- 자동 승인 범위를 넓히지 말고 반복 작업만 좁게 허용하라.
- 큰 작업은 plan, 끝날 때는 verify, 위험 작업은 review를 강제하라.
- 스타일 취향보다 데이터 무결성과 운영 안전성을 더 높은 레벨에 둬라.

## 9. 최종 평가

### 현재 평가

- 바이브코딩 활용도: 높음
- 에이전트 커스터마이징 수준: 높음
- 운영 일관성: 중간
- 규칙 구조 건전성: 중간 이하
- 자동화 안전성: 낮음~중간
- 앞으로의 개선 여지: 매우 큼

### 판정

당신은 에이전트를 엉뚱하게만 쓰는 사용자가 아니다.
오히려 너무 적극적으로 써서, 이제는 "많이 쓰는 단계"에서 "운영을 정리하는 단계"로 넘어가야 하는 상태다.

한마디로 정리하면:

"바이브코딩을 못 쓰는 게 아니라, 너무 많은 규칙과 자동화를 동시에 안고 있어서 다음 단계로 가려면 정리 작업이 먼저 필요하다."
